# Versioning policy (PokéRem)

## Canonical version for RemNote

The **RemNote plugin manifest** is the source of truth for what users and the marketplace see (including **semver**):

- File: [`public/manifest.json`](../public/manifest.json) — `version.major` / `version.minor` / `version.patch`, and **`author`** (plugin details page author line).

Keep **`author`** in sync with [`src/releaseMeta.ts`](../src/releaseMeta.ts) `POKEREM_AUTHOR` (in-app About / footer) and [`package.json`](../package.json) `author` when you change the displayed name.

The production build copies `public/` into `dist/`, so the submitted zip must contain an updated manifest.

**Plugin listing icon (RemNote only):** Ship both [`public/logo.svg`](../public/logo.svg) and [`public/logo.png`](../public/logo.png) at the **plugin bundle root** (see `remnote-plugin validate`). The desktop app’s plugin **card / details** UI is believed to prefer **PNG** (SVG may not be used for that surface even when validate reports “Logo: SVG”). Without usable raster art, RemNote shows an autogenerated initial (e.g. “P”). Regenerate PNG after tweaks: `npm run generate:logo` (requires Pillow). These files are **not** the right-sidebar tab image.

**Right-sidebar tab icon:** Still [`public/assets/items/poke-ball.png`](../public/assets/items/poke-ball.png) via `widgetTabIcon` in [`src/widgets/index.tsx`](../src/widgets/index.tsx) — same asset as shop/bag branding.

## In-app version label

[`src/releaseMeta.ts`](../src/releaseMeta.ts) exports `POKEREM_VERSION` for the Settings → About line **and** as a cache-bust query on the **right-sidebar tab icon** URL (`assets/items/poke-ball.png` in [`src/widgets/index.tsx`](../src/widgets/index.tsx)). Bump it whenever you bump the manifest
and `package.json` so the UI never shows a stale number, and so the tab picks up an updated ball PNG if that asset changes.

## npm `package.json` version

[`package.json`](../package.json) `version` is kept **in sync** with the manifest (same `MAJOR.MINOR.PATCH`) so:

- Developers see one number in the repo root.
- Release notes and tags can reference a single version string.

If they ever diverge by mistake, treat **manifest** as authoritative for the next marketplace upload and realign `package.json` and `releaseMeta.ts`.

**Automated check:** run **`npm run version:check`** (reads manifest, `package.json`, `package-lock.json` root, and `POKEREM_VERSION`). It also runs at the start of **`npm run release`**.

## Changelog

User-facing changes for each release should be recorded in [`CHANGELOG.md`](../CHANGELOG.md) (Keep a Changelog style). Bump the manifest (and package) **before** tagging or uploading a new zip.

## Pre-release builds

During active development, patch bumps can be frequent. That is fine; what matters is **monotonic increases** for published marketplace versions so RemNote can order updates correctly.
